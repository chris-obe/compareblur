# Agent Guidance

## Deployment

- Use `blur` as the visible product/site name in app chrome and browser titles; do not reintroduce `howmuchblur` as user-facing branding.
- For Christian's fork publishing flow, use the `fork` remote at `https://github.com/chris-obe/Blur.git` and push ongoing work to `fork/master` unless a separate feature branch or PR is explicitly requested.
- Keep the root README product-led and visually consistent with the app's restrained hard-edged style. It should introduce blur, its back story, and practical use cases before operational details; keep self-hosting/setup content minimal and collapsed near the bottom.
- Keep `AppShell` as the viewport owner for authenticated/app routes: sidebar, top bar, and bottom nav frame the app in a fixed viewport, and route content scrolls inside the shell's main region. Do not reintroduce document/body scrolling for app-shell routes; pages that need pinned subpanes should inherit a bounded `h-full min-h-0` layout from the shell instead of using ad hoc `100vh` calculations.
- Keep the desktop main navigation sidebar collapsible. New sidebar destinations must work in both expanded and icon-only states, with hard-edged hover/focus tooltips for collapsed labels. Keep the collapse/expand control in the sidebar's top brand chrome, not as a navigation item or footer control.
- Primary app screen availability must go through the shared runtime feature flag layer. New flaggable screens should add one known flag id, route guard, and navigation metadata entry rather than hiding routes or nav items with ad hoc local state.
- Keep Compare system selection as three minimal stacked options: a `From kit` dropdown containing full saved body/lens combinations, a catalog `Camera + lens` picker, and a `Manual` entry for format, focal length, and aperture. Reuse the app's custom dropdown/search components rather than native HTML select styling.
- Compare, Suggestions, and gallery/lightbox recommendation surfaces must reuse the shared look-matching, candidate-generation, taste-profile, and optics UI modules rather than adding one-off blur/FOV scoring, kit/catalog matching, or separate taste/reaction stores.
- Keep Compare's primary optical explanation in one graph: background blur by selected subject framing, with optional standing-distance/FOV context and selectable fixed-list or tracked-line readouts inside that graph. Do not add a separate framing-vs-blur visualisation to the Compare route.
- Keep Compare system rows dense and scalable: no hard cap on compared systems, source-aware non-`sys-*` labels, compact camera/format/crop metadata, focal/aperture fields, delete action, and one combined style menu that previews the adaptive line color/style output wired into the chart. Compare chart selectors/readouts should default to all systems where practical and use crop factor plus focal length/aperture labels rather than arbitrary system identifiers.
- Treat equivalence readouts as selected-target-format equivalents. When a user can choose a target format, label the result as `[selected format] Equivalent` and compute against that target instead of hard-coding full-frame language.
- Keep the React/Vite app deployable as a static Cloudflare Pages site from the repository root with build command `cd app && npm ci && npm run build` and output directory `app/dist`; the root `package.json` build script must continue to delegate to that same command so Pages projects configured as `bun run build` still work. Do not add Cloudflare Workers or Pages Functions for graph/blur generation unless a server-only requirement is introduced.
- For CMS or user-generated photo upload features, keep the Pages frontend static and handle mutable data through Cloudflare serverless services: Pages Functions or Workers for API/auth, R2 for image objects, and D1 for gallery/user metadata. Do not use GitHub commits and rebuilds as the publishing path for user uploads or generated gallery records. Gallery uploads should be browser-processed before R2 storage to a 2048px maximum long edge and 1 MiB maximum object size, with server-side API enforcement of the same size cap.
- Treat the public gallery as Cloudflare-backed data: D1 is the source of truth for photo metadata and moderation status, while R2 stores image objects. Do not add new gallery photos by editing `gallery.seed.ts` or committing image assets for normal publishing; use the gallery admin/API path so additions do not require app rebuilds. Gallery upload surfaces must reuse the shared EXIF/catalog metadata resolver rather than implementing parallel camera/lens guessing, and must not attach fuzzy camera/lens catalog IDs when EXIF and catalog data disagree; preserve the EXIF display text and leave catalog IDs empty until the catalog has a compatible exact match. Gallery records must store canonical engine format ids only, validated before D1 writes, and upload clients must canonicalize EXIF-detected crop/sensor formats through the shared gallery format resolver before sending API payloads. Public gallery rendering must degrade per-row rather than letting one malformed record break unrelated photos. Admin views should render already-approved gallery images through the public approved image route when possible, reserving protected admin image routes for non-public moderation states. Gallery edit/upload surfaces must reuse the shared catalog combobox and central gallery tag repository rather than adding raw camera/lens/tag inputs; catalog IDs are derived metadata and should be read-only in UI. Gallery and lightbox format dropdowns must reuse the shared gallery format option/label helpers and show crop factors consistently. Gallery uploads must store a required subject/framing distance preset, and Compare must consume that preset rather than defaulting every gallery image to the generic subject width. Public embed/photo metadata surfaces must reuse the shared gallery payload and optics/equivalence helpers, and must expose only curated metadata fields rather than raw sensitive EXIF. Embed routes and iframe-code dialogs must reuse `embedSnippet` URL/snippet helpers and `EmbedPhotoFrame`/`EmbedGalleryCard` render components; multi-photo embeds must remain public-only, approved/published-filtered, and capped rather than introducing stored ad hoc selected-set records.
- Persist gallery reactions in Cloudflare D1 behind Auth0-scoped API routes. The browser may keep a local fallback for anonymous use, but signed-in reaction writes must be keyed by Auth0 `sub`, aggregate counts must come from D1, and admin analytics must not expose mutation paths that can impersonate another user.
- Keep Auth0 and Cloudflare admin credentials out of the Vite client bundle and committed files. Auth0 SPA domain, client ID, and audience values used by `VITE_` variables are public browser configuration and may be committed as build-safe defaults when needed for static Pages builds; Auth0 Management API, Cloudflare API, R2, D1, and image administration secrets must live in local-only env files, Conductor local/user settings, or Cloudflare secrets. Do not commit shared admin tokens or token hashes in `wrangler.toml`, examples, docs, tests, fixtures, or generated artifacts.
- Keep `/admin` usable without authentication in local Vite development unless `VITE_ADMIN_REQUIRE_AUTH=true` is set, but require an Auth0 admin role/permission or stronger Cloudflare Access/backend enforcement in production. Treat frontend admin checks as UI gating only; admin APIs must enforce authorization server-side and must not rely on client-hidden routes.
- Route browser admin API calls through same-origin trusted endpoints, such as Cloudflare Pages Functions in production and Vite middleware in local development, so backend tokens can be injected server-side without exposing them to the Vite client.
- Keep Auth0 Management API usage behind Pages Functions or Workers. The frontend may display user/admin summaries returned by trusted endpoints, but must never call the Management API directly or receive Management API credentials/tokens.
- When `VITE_ADMIN_REQUIRE_AUTH=true` in local Vite development, proxy `/api/admin/*` through deployed Pages admin endpoints so local login tests exercise Auth0 JWT verification instead of silently falling back to development bypasses or local admin secrets.
- For account features in this personal-project portfolio, treat Auth0 as the shared identity layer when practical, but keep app-specific profiles, membership, entitlements, and CMS data in this app's own storage. Do not silently create or duplicate full app-local accounts for every cross-portfolio user; create local records lazily on first visit, explicit opt-in, or clear product need.
- Keep editable cross-portfolio profile basics in Auth0 `user_metadata`, but keep app activity such as gallery uploads, reactions, likes, moderation state, and ownership in this app's D1/R2-backed data. New gallery uploads must record the authenticated uploader when available; legacy or unattributed gallery rows are treated as admin-owned.
- Account-owned gallery and album APIs must enforce `submitted_by`/`owner_sub` server-side on every read and mutation. Private image routes must require Auth0 and be rendered in the UI via authenticated blob fetches, while public routes may only expose approved photos and published albums.
- Account album creation and editing should be album-first: the user names/describes the album, uploads or drops photos inside that album flow, then edits photo details in context. The `/albums` page should remain an album index with preview cards, route-backed album detail pages should support view/edit modes, and all-images browsing should be an icon action-bar toggle rather than a primary album selector. Do not add detached global bulk-upload controls that force users to reconcile uploads with albums afterward. On desktop, keep the album edit workspace as a bounded pane: album navigation stays pinned/stable while the main editor and options/actions columns scroll internally; mobile may remain stacked and page-scrollable.
- Album detail pages should keep a stable information order: breadcrumb back to `Albums`, then the album title row with visibility/view-mode/embed controls, then description, then the interaction bar. Do not reintroduce a separate standalone edit button above the interaction bar once the view/edit toggle exists inline with the title row.
- Album visibility should be presented to users as `Private`/`Public` even if storage still uses draft/published states internally. Public albums may carry an optional password, and password-protected albums should not be treated as embeddable until a dedicated protected-embed flow exists.
- Album photo selection surfaces should use custom select controls instead of raw browser checkboxes, support `Select all` for the current visible set, and support shift-click range selection across the current visible ordering.
- Public albums are owner-scoped public views, not entries in the shared gallery. Making an album public must not auto-promote its photos into the central public gallery, and making it private must not mutate gallery moderation state.
- Keep gallery moderation separate from album visibility. Use per-photo album membership visibility for public album inclusion, and reserve gallery approval state for the central public gallery and approved public-by-id image routes only.
- Keep blog embed template management separate from album CRUD. The blog-embed/settings surface may choose preview photos or published album context, but album creation and editing must live only in the `/albums` flow and the albums settings section.
- Photo lightbox surfaces must reuse the shared lightbox shell in `app/src/components/lightbox`; gallery-specific panels may customize metadata/actions, but should not recreate separate modal frame, keyboard, navigation, animation behavior, or metadata-card layout. Keep shared lightbox metadata compact, non-duplicative, and resilient to long camera/lens/format text.
- Blog/embed template controls must stay schema-synchronized across `app/src/lib/embedTemplate.ts`, `app/src/lib/galleryApi.ts`, `functions/_lib/embed.ts`, the Settings preview, public embed rendering, and generated iframe snippets. Keep image and multi-image/album embed modes separate in settings and rendering; multi-image defaults should favor a clean contact-sheet grid with one configurable, contrast-safe Open in blur action rather than repeated per-image metadata/actions. Frame options such as border width, curated border colour, and square-image cropping must be saved per mode and reflected in both previews and public embeds. Size controls should constrain the copied iframe footprint as well as the in-app preview.
- Do not install or maintain agent skills as workspace-local source files. Use the central Agent Skills vault at `/Users/christianobe/Library/Mobile Documents/iCloud~md~obsidian/Documents/Christian's Vault/98 🧩 Agent Skills`; if workspace tooling expects `.agents/skills`, make it a symlink to that central directory.
- Treat camera and lens catalog data as generated output from source adapters plus curated overrides. Do not add parallel hand-maintained camera or lens datasets in the app; add new sources or deterministic transforms under the catalog orchestration plan instead.
- When filling reported catalog inventory gaps, add records through the existing source adapters or curated override modules, then pin representative camera/lens IDs in `catalog/src/check.mjs` so future generated catalog rebuilds cannot silently drop that coverage.
- Catalog exports must include per-item provenance for every camera and lens. Source changes must update the recon/bakeoff report so external, derived, and curated records remain auditable, and curated records must state why no current external source covers them. Validate full catalog exports through the shared catalog validator before writing local fallback files, Worker R2 exports, or reports.
- Keep the catalog admin surface able to inspect the joined catalog export that the app consumes, plus the latest Cloudflare Worker/R2 export, so source joins, provenance, and compatibility gates can be verified without leaving the admin UI. Make this inspector table-first for bulk review, with raw JSON available as a secondary detail/debug view.
- Keep catalog refresh infrastructure in Cloudflare as a separate Worker/D1/R2 flow from the static Pages frontend. Use D1 settings for adjustable refresh cadence, R2 for source snapshots and generated exports, and admin endpoints for manual refresh/status rather than redeploying the app for catalog updates.
- Keep catalog admin controls dense and non-duplicative. Build settings should use compact, same-height responsive control groups, and diagnostic/provenance panels should be collapsed drilldowns from interactive summary tiles rather than always-visible duplicate cards. Catalog/admin inspector tables should stay semantic tables with sticky headers, internal scroll, compact top controls, and icon pagination rather than detached footer controls; dataset counts, source-type filters, view mode, pagination, and export actions should live in one consolidated explorer toolbar instead of separate card/tab/button surfaces.
- Keep large catalog JSON out of direct Vite imports. Serve one generated full catalog JSON as a runtime asset or Cloudflare export, and load it through `CatalogProvider` so dataset updates do not inflate the main app bundle. Do not add a separate app-local LensDB JSON surface.
- Keep gallery D1 schema migrations under `migrations/gallery`, and keep the root `wrangler.toml` D1 binding pointed at that directory so `wrangler d1 migrations apply` uses the same migration set as the app code.
